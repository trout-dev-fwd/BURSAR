# V2 Data Model — AI Accounting Assistant

This document defines all data storage changes for V2 Phase 1. V2 introduces three data stores beyond the existing SQLite entity databases: per-entity TOML configuration files, per-entity context markdown files, and a secrets file. It also adds one new SQLite table and one new column to an existing table.

---

## Design Decisions

### Decision 1: Conversation history is NOT persisted

AI chat conversation history lives in memory only and is lost when the application closes. This is intentional:
- Keeps the implementation simple (no serialization, no schema for messages)
- Conversations are transient — they serve an immediate purpose
- Important interactions are captured in the audit log (see Decision 2)
- The `/compact` command summarizes in-memory, not to disk

**Future enhancement (out of scope):** Session persistence could serialize message history to a file on close and reload on open. The data model does not preclude this.

### Decision 2: AI interactions logged to existing audit_log table

Every AI interaction produces two audit log entries:
- `AiPrompt` — the user's question as the description
- `AiResponse` — a single-line summary generated by Claude (NOT the full response)

The full response is displayed in the chat panel but not stored. The single-line summary is requested via the system prompt: Claude always returns both a full response and a `summary` field that gets logged.

**Why audit_log and not a new table:** The audit log already exists, is append-only, is included in the existing audit log tab UI, and Claude can query it via the tool system. Reusing it means AI interactions appear in the natural audit trail alongside JE posts, account changes, and other mutations. No new infrastructure needed.

**Why summary instead of full response:** Full AI responses can be 500+ words. Logging them inflates the database and wastes tokens when Claude reads its own history via the audit tool. A one-line summary captures the gist and is sufficient for Claude to reconstruct context (e.g., "Analyzed Q1 insurance expenses across accounts 5100-5300, recommended reviewing policy renewal timing").

### Decision 3: Learned import mappings in SQLite, not markdown

Transaction-to-account mappings learned during CSV import are stored in a new SQLite table (`import_mappings`) rather than the entity context markdown file.

**Why SQLite over markdown:**
- Mappings grow over time — could reach hundreds of entries after a year of monthly imports
- Pass 1 matching requires querying patterns (exact match, then substring) — SQL is naturally suited
- Avoids parsing markdown on every import
- Opens the door for a future mapping management UI
- Reduces token waste — Claude doesn't ingest the full mapping table via context file, it can query specific mappings via a tool

### Decision 4: Per-entity TOML configuration files

Entity-specific settings (AI persona, bank accounts, bank formats, last import directory) live in separate per-entity TOML files rather than nested inside `workspace.toml`.

**Why separate files:**
- `workspace.toml` stays lean and universal (global settings only)
- Each entity is self-contained — its bank formats, linked accounts, and preferences travel together
- Scales cleanly as entities are added (no ballooning workspace file)
- Bank formats are entity-scoped because the same bank name may map to different CoA accounts across entities (user churns bank accounts)

**File location:** Same directory as `workspace.toml`. Referenced by path in the entity's `workspace.toml` entry.

### Decision 5: Entity context markdown files remain for business context only

The per-entity `.md` context file stores free-form business context written by the user (or by Claude's suggestions). It does NOT store learned mappings (those are in SQLite) or configuration (that's in TOML).

**Contents:**
- Business description and notes
- Domain-specific knowledge for the AI persona
- Any user-written guidance for Claude

**Why keep this file at all (instead of putting everything in SQLite):** The context file is designed to be human-written and human-edited. It's the user's notes to their AI accountant. Putting free-form business context in a database column would make it invisible and hard to edit. Markdown is the right format for prose.

### Decision 6: API key in a separate secrets file

The Anthropic API key lives in `~/.config/bookkeeper/secrets.toml`, completely separate from project configuration.

**Why not workspace.toml:** The workspace file likely lives in or near a git repo. API keys in version-controlled files is a common leak vector.

**Why not environment variable:** Environment variables work but are invisible and easy to forget. A dedicated secrets file with a well-known path is explicit, discoverable (error messages can point to the exact file), and follows the XDG config convention.

## SQLite Schema Changes

### New Table: import_mappings

Stores learned transaction-to-account mappings from CSV imports. Each entity's database gets this table.

```sql
CREATE TABLE IF NOT EXISTS import_mappings (
    id INTEGER PRIMARY KEY,
    description_pattern TEXT NOT NULL,
    account_id INTEGER NOT NULL REFERENCES accounts(id),
    match_type TEXT NOT NULL CHECK(match_type IN ('exact', 'substring')),
    source TEXT NOT NULL CHECK(source IN ('confirmed', 'ai_suggested')),
    bank_name TEXT NOT NULL,
    created_at TEXT NOT NULL,
    last_used_at TEXT NOT NULL,
    use_count INTEGER NOT NULL DEFAULT 1,
    UNIQUE(description_pattern, bank_name)
);
```

**Columns:**

| Column | Type | Purpose |
|--------|------|---------|
| `id` | INTEGER PK | Auto-increment primary key |
| `description_pattern` | TEXT NOT NULL | The bank statement description to match against (e.g., "WELLS FARGO MORTGAGE") |
| `account_id` | INTEGER NOT NULL FK | The CoA account this description maps to |
| `match_type` | TEXT NOT NULL | How to match: `exact` (full string equality) or `substring` (pattern contained in description) |
| `source` | TEXT NOT NULL | How this mapping was created: `confirmed` (user explicitly confirmed) or `ai_suggested` (Claude suggested, user didn't reject) |
| `bank_name` | TEXT NOT NULL | Which bank account this mapping applies to (e.g., "SoFi Checking") — mappings are bank-scoped because the same description may mean different things from different banks |
| `created_at` | TEXT NOT NULL | ISO 8601 timestamp of when the mapping was first created |
| `last_used_at` | TEXT NOT NULL | ISO 8601 timestamp of the most recent successful match — useful for identifying stale mappings |
| `use_count` | INTEGER NOT NULL DEFAULT 1 | Number of times this mapping has been used — helps gauge reliability |

**Unique constraint:** `(description_pattern, bank_name)` — the same description from the same bank always maps to one account. If a mapping needs to change, the existing row is updated.

**Query patterns:**

Pass 1 matching (ordered by priority):
```sql
-- Exact match (highest priority)
SELECT account_id FROM import_mappings
WHERE bank_name = ?1 AND match_type = 'exact' AND description_pattern = ?2;

-- Substring match (second priority)
SELECT account_id, description_pattern FROM import_mappings
WHERE bank_name = ?1 AND match_type = 'substring' AND ?2 LIKE '%' || description_pattern || '%'
ORDER BY LENGTH(description_pattern) DESC;
```

The substring query orders by pattern length descending so longer (more specific) patterns take priority over shorter ones.

Update on use:
```sql
UPDATE import_mappings SET last_used_at = ?1, use_count = use_count + 1 WHERE id = ?2;
```

### Modified Table: journal_entries

Add one nullable column for import traceability.

```sql
ALTER TABLE journal_entries ADD COLUMN import_ref TEXT;
```

**Migration:** Follow the existing `EntityDb::open()` pattern — check `PRAGMA table_info('journal_entries')` for the column, add if missing. This is the same pattern used for `accum_depreciation_account_id` on `fixed_asset_details`.

**Column:** `import_ref TEXT` — nullable. Contains a composite reference string for tracing a draft back to its source bank statement line.

**Format:** `"{bank_name}|{date}|{description}|{amount}"`

Example: `"SoFi Checking|2026-03-15|WELLS FARGO MORTGAGE|-1247.32"`

**Used for:**
- **Duplicate detection:** Hash `import_ref` values against existing entries from the last 90 days before importing
- **Traceability:** User can see which statement line generated a draft
- **Re-matching:** Query for drafts with `import_ref IS NOT NULL` and incomplete lines (one side blank) to find candidates for Shift+U batch re-match

**Query patterns:**

Duplicate detection:
```sql
SELECT import_ref FROM journal_entries
WHERE import_ref IS NOT NULL
AND created_at >= date('now', '-90 days');
```

Incomplete imports for re-matching:
```sql
-- Drafts with import_ref that have fewer than 2 complete lines
SELECT je.id, je.import_ref
FROM journal_entries je
WHERE je.status = 'Draft'
AND je.import_ref IS NOT NULL
AND (
    SELECT COUNT(*) FROM journal_entry_lines jel
    WHERE jel.journal_entry_id = je.id
    AND jel.account_id IS NOT NULL
) < 2;
```

### Modified Table: audit_log (new action types only)

No schema change needed — the existing `audit_log` table structure supports new action types via the `action TEXT` column.

**New action types:**

| Action | Description Format |
|--------|-------------------|
| `AiPrompt` | The user's question verbatim (truncated to 500 chars if longer) |
| `AiResponse` | Claude's single-line summary of its response |
| `AiToolUse` | "Used {tool_name}({key_params})" — e.g., "Used get_account(5100)" |
| `CsvImport` | "Imported {N} transactions from {bank_name}: {M} matched, {K} AI-matched, {J} manual" |
| `MappingLearned` | "Learned: '{description}' → {account_number} ({account_name}) [{source}]" |

These follow the existing pattern where `action` is a categorization tag and `description` is a human-readable detail string.

## Configuration File Schemas

### workspace.toml (universal settings)

```toml
# Existing fields (unchanged)
report_output_dir = "~/accounting/reports"

# New fields
context_dir = "~/accounting/context"

# New section
[ai]
persona = "Professional Tax Accountant specializing in small business LLCs"
model = "claude-sonnet-4-20250514"    # optional, defaults to claude-sonnet-4-20250514

# Existing structure with new field
[[entities]]
name = "Acme Land LLC"
db_path = "~/accounting/database/acme_land.sqlite"
config_path = "acme_land.toml"          # NEW: relative to workspace.toml directory, or absolute with tilde
```

**New fields:**

| Field | Required | Default | Purpose |
|-------|----------|---------|---------|
| `context_dir` | No | None (AI features disabled without it) | Directory for per-entity context markdown files |
| `[ai].persona` | No | `"Professional Tax Accountant"` | Default AI persona for all entities |
| `[ai].model` | No | `"claude-sonnet-4-20250514"` | Claude model to use |
| `entities[].config_path` | No | None (entity uses global defaults only) | Path to per-entity TOML config |

**Backwards compatibility:** All new fields are optional. An existing V1 `workspace.toml` works without modification. AI features are gracefully unavailable without the `[ai]` section and `context_dir`.

### Per-Entity TOML (e.g., acme_land.toml)

```toml
# Entity-specific AI persona (overrides global [ai].persona)
ai_persona = "Real estate tax specialist focused on residential rental properties in Texas"

# Last directory used for CSV import (auto-updated on each import)
last_import_dir = "~/Downloads"

# Bank account configurations
[[bank_accounts]]
name = "SoFi Checking"
linked_account = "1100"
date_column = "Date"
description_column = "Description"
amount_column = "Amount"
debit_is_negative = true
date_format = "%m/%d/%Y"

[[bank_accounts]]
name = "Alliant Checking"
linked_account = "1110"
date_column = "Date"
description_column = "Description"
debit_column = "Debit"
credit_column = "Credit"
date_format = "%m/%d/%Y"

[[bank_accounts]]
name = "Chase Credit Card"
linked_account = "2300"
date_column = "Trans Date"
description_column = "Description"
amount_column = "Amount"
debit_is_negative = false
date_format = "%m/%d/%Y"
```

**Fields:**

| Field | Required | Purpose |
|-------|----------|---------|
| `ai_persona` | No | Override global AI persona for this entity |
| `last_import_dir` | No | Auto-updated on each CSV import |
| `bank_accounts[].name` | Yes | Display name and identifier for this bank account |
| `bank_accounts[].linked_account` | Yes | CoA account number (TEXT) this bank maps to |
| `bank_accounts[].date_column` | Yes | CSV column name containing the transaction date |
| `bank_accounts[].description_column` | Yes | CSV column name containing the description/memo |
| `bank_accounts[].amount_column` | Conditional | CSV column name for single-amount format (mutually exclusive with debit_column/credit_column) |
| `bank_accounts[].debit_column` | Conditional | CSV column name for debit amounts in split format |
| `bank_accounts[].credit_column` | Conditional | CSV column name for credit amounts in split format |
| `bank_accounts[].debit_is_negative` | No (default true) | In single-amount format: if true, negative values are debits/withdrawals |
| `bank_accounts[].date_format` | Yes | `chrono` format string for parsing dates (e.g., `%m/%d/%Y`, `%Y-%m-%d`) |

**Design note:** Bank format configuration and linked account are unified in a single `[[bank_accounts]]` entry rather than separated into `[[bank_formats]]` + `[[bank_accounts]]`. This simplifies the config — each entry is a complete bank account definition. If the same bank format is used across entities, the column mapping is duplicated, which is acceptable given the expected scale (under 10 bank accounts per entity).

### ~/.config/bookkeeper/secrets.toml

```toml
anthropic_api_key = "sk-ant-..."
```

Single field. Auto-created directory on first AI use. Read lazily (not at startup).

### Per-Entity Context File (e.g., ~/accounting/context/acme_land_llc.md)

```markdown
# Acme Land LLC — AI Context

## Business Context
<!-- Describe your business here for better AI assistance -->
```

**Auto-created** on first AI interaction for this entity. The skeleton includes only a header and a placeholder section. The user fills in business context over time. Claude may suggest additions during conversations.

**No longer contains learned mappings** — those moved to the `import_mappings` SQLite table (see Decision 3).

**Path derivation:** Entity name → lowercase → spaces to underscores → strip non-alphanumeric except underscores → append `.md`. Example: "Acme Land LLC" → `acme_land_llc.md`.

## Integrity Invariants

### import_mappings
1. `description_pattern` + `bank_name` is unique — one mapping per description per bank
2. `account_id` references a valid account in `accounts` — foreign key enforced
3. `match_type` is constrained to `exact` or `substring` — CHECK constraint
4. `source` is constrained to `confirmed` or `ai_suggested` — CHECK constraint
5. `use_count` is always ≥ 1 (initialized to 1 on creation)
6. `last_used_at` is always ≥ `created_at`

### journal_entries.import_ref
1. Nullable — only present on imported transactions
2. Format is always `"{bank_name}|{date}|{description}|{amount}"` — enforced by application code, not database constraint
3. Not unique — theoretically a bank could produce identical transactions (same date, description, amount), though duplicate detection mitigates this

### audit_log (AI entries)
1. `AiPrompt` and `AiResponse` always appear as a pair — if the API call fails, only `AiPrompt` is logged (no response entry)
2. `AiToolUse` entries appear between an `AiPrompt` and its `AiResponse` — they document the intermediate tool calls
3. Description for `AiPrompt` is truncated to 500 characters if the user's question is longer
4. Description for `AiResponse` is always a single-line summary, never the full response

### Configuration files
1. `workspace.toml` is never auto-modified by the application — only `entities[].config_path` references are read
2. Per-entity TOML files may be auto-modified: `last_import_dir` is updated on import, new `[[bank_accounts]]` entries are appended during new bank setup
3. `secrets.toml` is read-only from the application's perspective
4. Context `.md` files may be auto-created (skeleton) but are otherwise only appended to (never overwritten or truncated)
