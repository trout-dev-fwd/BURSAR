# V2.2 Feature Specification — Distribution & Feedback

## Overview

V2.2 adds the infrastructure to distribute Bursar as a public application: automated
cross-platform releases, self-updating binaries, and in-app issue reporting. These
features transform Bursar from a personal tool into software that can be shared with
other users.

**What changes for the user**: Bursar checks for updates on launch. If a newer version
is available, it downloads and installs it automatically, then restarts. Users can
report bugs and request features directly from the help overlay without leaving the
terminal.

**What changes for the developer**: Pushing a git tag triggers a GitHub Actions workflow
that builds release binaries for Linux and Windows, generates SHA256 checksums, and
publishes a GitHub Release with all assets attached. No manual build-and-upload step.

---

## Success Criteria

The version is "done" when all of the following are true:

- [ ] All code references point to `trout-dev-fwd/bursar` (no references to `trout-accounting`)
- [ ] Version number on the splash screen is centered beneath the BURSAR logo
- [ ] Pushing a version tag (e.g., `v0.2.2`) triggers GitHub Actions to build Linux and Windows
      binaries and publish a GitHub Release with attached assets and SHA256 checksums
- [ ] On launch, Bursar checks GitHub Releases API for a newer version
- [ ] If an update is available, the splash screen shows an ASCII progress bar while downloading
- [ ] Downloaded binary is verified against the SHA256 checksum before installation
- [ ] After successful update, Bursar restarts into the new version automatically
- [ ] If the update fails (no internet, checksum mismatch, write error), Bursar falls through
      gracefully and launches the current version with a warning on the startup screen
- [ ] `b` key in the `?` help overlay opens a bug report modal
- [ ] `f` key in the `?` help overlay opens a feature request modal
- [ ] Submitting a report opens the user's browser with a pre-filled GitHub issue including
      the user's description, Bursar version, OS/platform, and last 5 audit log entries (bugs only)
- [ ] The `?` help overlay has a "Feedback" section at the bottom matching existing formatting
- [ ] `cargo fmt && cargo clippy -D warnings && cargo test` all pass with zero warnings

## Feature 2: Auto-Update with Self-Restart

### What

On every launch, Bursar checks whether a newer release exists on GitHub. If one does, it
downloads the platform-appropriate binary, verifies its SHA256 checksum, replaces the running
binary, and restarts. The user sees a progress bar on the splash screen during download.

### User-Visible Flow

1. App launches → splash screen appears with centered BURSAR logo and version number
2. HTTP GET to `https://api.github.com/repos/trout-dev-fwd/bursar/releases/latest`
   (with `User-Agent: bursar/{version}` header — GitHub returns 403 without one)
3. **No internet / API error** → skip silently, proceed to startup screen
4. **Current version is latest** → proceed to startup screen (no message needed)
5. **Newer version available** →
   - **Pre-flight checks**: verify binary is not a symlink and parent directory is writable.
     If either fails, fall through with warning (no download attempted)
   - Splash screen shows: `Updating to vX.Y.Z. . .`
   - Progress bar appears below: `▰▰▰▰▱▱▱▱▱▱ [40%]`
   - Download the platform-appropriate binary asset
   - Download `checksums.txt` asset
   - Verify SHA256 of downloaded binary against checksums.txt
   - **Verification passes** → replace binary, restart
   - **Verification fails** → show warning on startup screen, launch current version

### Technical Design

#### HTTP Requirements

All GitHub API requests (release check, asset downloads, checksum download) must include:
- `User-Agent: bursar/{version}` header (GitHub returns 403 without one)
- `Accept: application/vnd.github+json` header for API calls
- `Accept: application/octet-stream` header for asset downloads

#### Version Comparison

The running binary's version comes from `env!("CARGO_PKG_VERSION")` (set at compile time).
The latest release version comes from the GitHub API response's `tag_name` field (strip
leading `v` if present). Compare using semver: if remote > local, update is available.

New dependency: `semver` crate for version parsing and comparison.

#### Platform Detection

At compile time, use `cfg!(target_os)` and `cfg!(target_arch)` to determine the expected
asset filename:

| Platform | Asset Name |
|----------|------------|
| Linux x86_64 | `bursar-linux-x86_64` |
| Windows x86_64 | `bursar-windows-x86_64.exe` |

The GitHub Release assets are matched by name from the API response's `assets` array.

#### Pre-flight Checks

Run after confirming an update is available, but **before** downloading anything:

1. `std::env::current_exe()` → resolved path of the running binary
2. **Symlink check:** compare the resolved path against the original path. If the binary
   was launched through a symlink, the rename would break the symlink. Fall through with
   warning: `"Update failed: binary is a symlink. Replace it manually."`
3. **Write permission check:** attempt to create and immediately delete a temp file in the
   binary's parent directory. If no write access, fall through with warning: `"Update
   failed: permission denied on {dir}. Try running with appropriate permissions or moving
   the binary to a user-writable location."`

These checks avoid wasting bandwidth on a download that can't be installed.

#### Download with Progress

Use `ureq` to download the binary. The `Content-Length` header provides the total size for
the progress bar. Read the response body in chunks (e.g., 8 KB), writing to a temporary
file and updating the progress bar after each chunk.

If the `Content-Length` header is missing (chunked transfer encoding), show an indeterminate
progress animation instead of a percentage:

```
              Updating to v0.2.2. . .
        ▰▱▰▱▰▱▰▱▰▱
```

The indeterminate bar alternates `▰▱` across 10 characters (static, not animated — animation
would require async). Once the download completes, the bar fills to `▰▰▰▰▰▰▰▰▰▰ [100%]`
briefly before the restart. GitHub typically includes `Content-Length` on release assets, so
the indeterminate path is a fallback, not the common case.

The progress bar renders on the splash screen below the logo. Format:

```
              Updating to v0.2.2. . .
        ▰▰▰▰▱▱▱▱▱▱ [40%]
```

The characters: `▰` (U+25B0) for filled, `▱` (U+25B1) for unfilled. Bar width: 10
characters (each represents 10%). The `. . .` trailing dots on the status line are
literal (not animated). Percentage is `bytes_received / content_length * 100`.

Since this is a synchronous app with no async runtime, the download blocks the event loop.
This is acceptable because it happens before the main UI loads and the progress bar updates
happen between chunk reads via direct `terminal.draw()` calls (same pattern as the forced
render before AI API calls in v2).

#### SHA256 Verification

Download `checksums.txt` from the same release. Expected format (one line per asset):

```
a1b2c3d4...  bursar-linux-x86_64
e5f6g7h8...  bursar-windows-x86_64.exe
```

Parse the file, find the line matching the current platform's asset name, extract the
expected hash. Compute SHA256 of the downloaded binary file. Compare. If mismatch, delete
the downloaded file and fall through.

New dependency: `sha2` crate (from the RustCrypto project) for SHA256 computation.

#### Binary Replacement

**Linux:**
1. `std::env::current_exe()` → path of the running binary (e.g., `/home/user/bin/bursar`)
2. Downloaded binary is at a temp path alongside it (e.g., `/home/user/bin/bursar.new`)
3. Rename current binary: `bursar` → `bursar.old`
4. Rename new binary: `bursar.new` → `bursar`
5. Set executable permission on the new binary (`std::fs::set_permissions`, mode `0o755`)
6. `std::os::unix::process::Command::exec()` to replace the current process with the new binary
   (passes the same args, if any)

**Windows:**
1. Same steps 1-4, but Windows allows renaming a running `.exe` (just not deleting it)
2. No permission step needed (Windows doesn't use Unix file permissions)
3. Cannot use `exec()` (Unix-only). Instead: explicitly restore the terminal (leave alternate
   screen, disable raw mode) to avoid leaving the terminal in a broken state, then
   `std::process::Command::new(new_path).spawn()` to launch the new binary, then
   `std::process::exit(0)` to terminate the old one

**Cleanup:** On startup (before the update check), look for and delete `bursar.old` /
`bursar.old.exe` next to the current binary. Silent failure if the file doesn't exist or
can't be deleted.

#### Failure Handling

Any failure in the update pipeline results in a graceful fallthrough:

| Failure Point | Behavior |
|---------------|----------|
| No internet / DNS failure | Skip update check, proceed normally |
| GitHub API returns non-200 / 403 | Skip update check, proceed normally |
| Asset not found for platform | Skip, proceed normally |
| Binary is a symlink | Show warning: "binary is a symlink. Replace it manually." |
| No write permission on binary dir | Show warning: "permission denied on {dir}" |
| Download interrupted | Delete partial file, show warning on startup screen |
| No Content-Length header | Download with indeterminate progress indicator |
| Checksum mismatch | Delete downloaded file, show warning on startup screen |
| Cannot rename current binary | Show warning on startup screen |
| Cannot rename new binary into place | Attempt to restore old binary, show warning |
| `exec` / spawn fails | Show warning (user will need to restart manually) |

Warning format on startup screen: `"Update to vX.Y.Z failed: {reason}. Running v{current}."`
This displays as the `update_notice` field on `StartupScreen`, same mechanism as the existing
update-available notice.

### Splash Screen Changes

- **Center the version number** beneath the BURSAR ASCII art logo (currently right-aligned)
- **Add progress bar area** below the version number, used only during active download
- **Minimum display time**: keep the existing 1-second minimum splash, but if an update is
  downloading, the splash stays visible until the download completes (no minimum needed —
  the download itself takes time)

### GitHub Actions CI/CD

A new workflow file at `.github/workflows/release.yml`:

**Trigger:** Push of a tag matching `v*` (e.g., `v0.2.2`, `v0.2.3`)

**Jobs:**

1. **build-linux**
   - Runner: `ubuntu-latest`
   - Steps: checkout, install Rust stable, `cargo fmt --check`, `cargo clippy --all-targets
     --all-features -- -D warnings`, `cargo test`, `cargo build --release`, rename binary to
     `bursar-linux-x86_64`
   - Upload artifact

2. **build-windows**
   - Runner: `windows-latest`
   - Steps: checkout, install Rust stable, `cargo fmt --check`, `cargo clippy --all-targets
     --all-features -- -D warnings`, `cargo test`, `cargo build --release`, rename binary to
     `bursar-windows-x86_64.exe`
   - Upload artifact

3. **release** (depends on build-linux, build-windows)
   - Runner: `ubuntu-latest`
   - Steps: download all artifacts, generate `checksums.txt` via `sha256sum`, create
     GitHub Release using `softprops/action-gh-release` (or equivalent), attach all
     three files (linux binary, windows binary, checksums.txt)

**Release notes:** Auto-generated from commit messages between the previous tag and the
new tag. Can be manually edited after the release is published.

**Tag convention:** `vMAJOR.MINOR.PATCH` matching the version in `Cargo.toml`. The `0.`
prefix indicates pre-1.0 (no stability guarantees). Bump to `v1.0.0` when Bursar is
feature-complete for public release. The developer is responsible for bumping the version
in `Cargo.toml` before tagging.

### Configuration

The `[updates]` section in `workspace.toml` already exists:

```toml
[updates]
github_repo = "trout-dev-fwd/bursar"   # default if not specified
```

If this key is missing or empty, the default `"trout-dev-fwd/bursar"` is used. This allows
forks to point to their own repo for updates.

| `sha2` | SHA256 checksum computation | Part of RustCrypto, well-maintained |

## Feature 3: In-App Feedback (Bug Reports & Feature Requests)

### What

Users can submit bug reports and feature requests from within the `?` help overlay.
The workflow collects user input in a modal, then opens the default browser with a
pre-filled GitHub issue.

### User-Visible Flow

**Bug report (`b` key in `?` overlay):**
1. User presses `?` to open help overlay
2. User presses `b`
3. Help overlay closes, a multi-line text input modal appears: "Describe the bug:"
4. User types their description, presses `Ctrl+S` to submit (or `Esc` to cancel)
5. Bursar constructs a GitHub issue URL with:
   - Title: `[Bug] {first line of user's description}`
   - Body: user's full description + system info block + last 5 audit log entries
   - Label: `bug`
6. Opens the URL in the default browser (`xdg-open` on Linux, `cmd /c start` on Windows)
7. Status bar shows: `"Bug report opened in browser"`

**Feature request (`f` key in `?` overlay):**
1-4. Same as bug report, but the modal says: "Describe the feature:"
5. Bursar constructs a GitHub issue URL with:
   - Title: `[Feature] {first line of user's description}`
   - Body: user's full description + system info block (no audit log for features)
   - Label: `enhancement`
6-7. Same as bug report

### URL Construction

GitHub supports pre-filled issues via query parameters:

```
https://github.com/trout-dev-fwd/bursar/issues/new?title={}&body={}&labels={}
```

All values must be URL-encoded (percent-encoding). The `body` parameter contains:

**Bug report body template:**
```
{user_description}

---

**System Information**
- Bursar version: {version}
- OS: {os} {arch}
- Entity: {active_entity_name or "None (startup screen)"}

**Recent Audit Log**
```
{last_5_audit_entries_formatted}
```
```

**Feature request body template:**
```
{user_description}

---

**System Information**
- Bursar version: {version}
- OS: {os} {arch}
```

System info values:
- `version`: `env!("CARGO_PKG_VERSION")`
- `os`: `std::env::consts::OS` (e.g., "linux", "windows")
- `arch`: `std::env::consts::ARCH` (e.g., "x86_64")
- `active_entity_name`: from `App.entity.name` if in Running state, otherwise "None"

Audit log entries: the 5 most recent entries from `audit_log` table, formatted as:
`{created_at} | {action_type} | {description}` — one per line. If no entity is open
(feedback from startup screen), this section is omitted.

#### URL Length Considerations

Browsers generally support URLs up to ~8,000 characters. The system info block is ~150
characters. Five audit log entries at ~100 characters each adds ~500. That leaves ~7,000
characters for the user's description after URL encoding (which roughly doubles the length),
so the practical user input limit is ~3,500 characters. This is generous for a bug report.

If the total URL exceeds 8,000 characters, truncate the user description with a note:
`"[Description truncated for URL length — please add details in the issue]"`

### Multi-Line Text Input Modal

A new widget for collecting multi-line text input. Distinct from the existing
`TextInputModal` which is single-line.

**Layout:** Centered modal, ~60% terminal width, ~40% terminal height. Title bar shows
"Describe the bug:" or "Describe the feature:". Text area below with basic editing.
Bottom bar shows: `Ctrl+S: Submit | Esc: Cancel`

**Editing capabilities (minimal):**
- Character input and backspace
- Enter for new lines
- Arrow keys for cursor movement (up/down/left/right)
- Home/End for line start/end
- Basic line wrapping (visual only, no word-wrap — wraps at widget boundary)

**No need for:** clipboard support, undo/redo, text selection, scroll indicators. This is
a quick-input field, not a text editor. If someone has a lengthy report they'll finish
composing it in the browser.

### Opening the Browser

**Linux:** `std::process::Command::new("xdg-open").arg(&url).spawn()`

**Windows:** `std::process::Command::new("cmd").args(["/c", "start", "", &url]).spawn()`

Note the empty string argument before `&url` on Windows — `start` treats the first quoted
argument as a window title, so the empty string prevents the URL from being consumed as the
title.

Fire-and-forget: spawn the process and don't wait for it. If the spawn fails (e.g.,
`xdg-open` not installed), show a status bar error: `"Could not open browser. URL copied
to clipboard."` — though clipboard access from a TUI is platform-dependent and may not be
reliable, so the fallback message should say: `"Could not open browser. Please file the
issue manually at https://github.com/trout-dev-fwd/bursar/issues"`

### Help Overlay Changes

The `?` help overlay (rendered in `src/app/key_dispatch.rs`) gains a new section at the
bottom:

```
  Feedback
    b             Report bug
    f             Request feature
```

This follows the exact same formatting as the existing "Global" and "Tab-specific" sections:
section header in orange/yellow, keys in cyan, descriptions in white, same column alignment.

### Key Dispatch Changes

The `b` and `f` keys are handled **only when the help overlay is visible** (`show_help == true`).
They are not global hotkeys — they live inside the help overlay's key handler. When pressed:

1. Set `show_help = false` (close the overlay)
2. Open the feedback modal with the appropriate type (Bug or Feature)

This means `b` and `f` retain their normal per-tab behavior when the help overlay is not
showing. No conflicts with existing hotkeys.

**Startup screen limitation:** The `?` help overlay only exists in the Running state (`App`),
not on the `StartupScreen`. Feedback is therefore only available after opening an entity.
This is intentional — the startup screen is a simple entity picker and doesn't warrant its
own help overlay. Users who encounter issues before opening an entity can file issues via
the GitHub web interface directly.

### State Management

New fields on `App`:

```rust
feedback_modal: Option<FeedbackModal>,  // Multi-line input modal
```

`FeedbackModal` is a struct containing:
- `feedback_type: FeedbackType` — enum: `Bug`, `Feature`
- `text_buffer: Vec<String>` — lines of text
- `cursor_row: usize`
- `cursor_col: usize`

The feedback modal is checked in the key dispatch priority chain, slotting in at the same
level as other modal overlays (above tab dispatch, below user guide and help overlay).

### New Files

| File | Purpose |
|------|---------|
| `src/widgets/feedback_modal.rs` | Multi-line text input + URL construction + browser launch |

### New Enums / Types

| Type | Variants | Location |
|------|----------|----------|
| `FeedbackType` | `Bug`, `Feature` | `src/widgets/feedback_modal.rs` |
| `FeedbackAction` | `Submit(FeedbackType, String)`, `Cancel` | `src/widgets/feedback_modal.rs` |

## Constraints & Gotchas

### Platform-Specific

- **`exec()` is Unix-only.** The restart path must be `cfg`-gated: `exec` on Linux,
  spawn-then-exit on Windows.
- **File permissions are Unix-only.** The `chmod` step after binary replacement must be
  `cfg`-gated to Linux only.
- **Windows terminal cleanup before restart.** On Windows, `std::process::exit(0)` may
  bypass Rust's drop guards. Explicitly restore the terminal (leave alternate screen,
  disable raw mode) *before* spawning the new process and exiting. On Linux, `exec()`
  replaces the process and the new binary re-initializes the terminal, so this isn't
  needed.
- **`xdg-open` may not exist** on minimal Linux installs. The browser-open function should
  handle spawn failure gracefully.
- **Windows `start` command** needs an empty title argument before the URL to avoid the URL
  being interpreted as a window title.

### Update Safety

- **Never delete the old binary before confirming the new one is in place.** The rename
  sequence (old → `.old`, new → target) ensures there's always a working binary on disk.
- **SHA256 verification is mandatory.** Never install an unverified binary. A corrupted
  binary could brick the app.
- **Cleanup of `.old` file happens on next launch**, not during the current session. This
  avoids any risk of deleting the wrong file during the replace sequence.
- **Symlinks break the rename sequence.** `std::env::current_exe()` resolves through
  symlinks. If the binary is symlinked (e.g., `/usr/local/bin/bursar` →
  `/home/user/builds/bursar`), renaming at the resolved path leaves the symlink pointing
  at `bursar.old`. Detect this and skip the update with a descriptive warning.
- **Write permissions on the binary directory.** If the binary lives in `/usr/local/bin/`
  or another root-owned directory, the rename will fail with a permission error. Test
  write access before attempting the download to avoid wasting bandwidth, and produce a
  clear error message suggesting the user either runs with appropriate permissions or
  moves the binary.

### GitHub API

- **User-Agent header is mandatory.** GitHub returns `403 Forbidden` on API requests
  without a `User-Agent` header. All requests must include `User-Agent: bursar/{version}`.
  This applies to both the release check and the asset downloads.

### URL Encoding

- The GitHub issue URL body must be fully percent-encoded. Characters like `#`, `&`, `+`,
  spaces, and newlines all need encoding. Use the `urlencoding` crate
  (`urlencoding::encode(&str) -> Cow<str>`) — do not roll a manual encoder.
- **Newlines in the body** must be encoded as `%0A` (not `%0D%0A`).

### Synchronous Download Blocking

- The download blocks the event loop. This is acceptable on the splash screen (no user
  interaction needed), but the progress bar must update between chunk reads. Each chunk
  read → update progress state → call `terminal.draw()` → read next chunk.
- If the download is very slow, the app appears frozen except for the progress bar. This
  is the expected behavior for a forced update on a synchronous app.

### Existing `update.rs` Behavior

- The current `update.rs` (72 lines) does a lightweight check and sets `update_notice` on
  the startup screen. V2.2 replaces this with the full download-and-restart flow. The
  existing "update available" notice path becomes the fallback for when download fails.

| Service | Used By | Failure Mode |
|---------|---------|-------------|
| GitHub Releases API | Update check + download | Skip update, launch current version |
| GitHub Issues (browser) | Bug reports + feature requests | Show error, suggest manual filing |

## Out of Scope (V2.2)

- **macOS builds** — can be added to the Actions workflow later when there's demand
- **ARM builds** (e.g., Raspberry Pi, Apple Silicon) — same, add when needed
- **Delta updates** — always downloads the full binary, no patch/diff system
- **Rollback to previous version** — if a new version has issues, the user can manually
  replace the binary; no in-app downgrade mechanism
- **GitHub authentication for issue creation** — using browser-based pre-filled URLs instead
- **Clipboard integration** — not reliably available from TUI across platforms
- **Issue templates on GitHub** — can be added to the repo independently of this feature;
  the pre-filled URL parameters work with or without templates
- **Automatic crash reporting** — users must actively choose to report issues

## Implementation Order

These features should be built in this order due to dependencies:

1. **Repo rename + splash screen polish** — foundational, no new dependencies, touches
   files that the other features also modify
2. **GitHub Actions workflow** — needed before auto-update can be tested against real releases
3. **Auto-update** — depends on the Actions workflow producing release assets
4. **In-app feedback** — independent of the other features, can be built last

Detailed phase breakdowns with task-level specifications will be created as separate
`v2.2-phase-*.md` files following the established pattern.
