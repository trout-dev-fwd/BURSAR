# V2.2 Phase 2: Auto-Update — Download, Verify, Replace, Restart

## Overview

This phase implements the full auto-update pipeline: checking GitHub for new releases,
downloading the binary with a progress bar, verifying the SHA256 checksum, replacing the
running binary, and restarting the application. This is the most complex phase in V2.2
due to platform-specific behavior and the need for robust failure handling.

## Critical Constraints

These constraints apply to EVERY task in this phase. Read them before starting.

1. **No async. No threads.** All HTTP calls are blocking via `ureq`. Progress bar updates
   happen between chunk reads via direct `terminal.draw()` calls.

2. **Every GitHub API request must include `User-Agent: bursar/{version}`.** GitHub returns
   403 without it. This is the #1 most common bug when working with the GitHub API.

3. **No `.unwrap()` in production code.** Every HTTP call, file operation, and JSON parse
   must use `?` propagation or explicit error handling. Update operations fail gracefully —
   they never panic.

4. **Platform-specific code uses `cfg` attributes**, not runtime checks. Use
   `#[cfg(target_os = "linux")]` and `#[cfg(target_os = "windows")]` to gate
   platform-specific blocks.

5. **The old binary is NEVER deleted during the update.** It is renamed to `.old` and
   cleaned up on the next launch. This ensures a working binary always exists on disk.

## Tasks

### Task 1: Rewrite Update Module — Types and GitHub API Client

**What:** Replace the existing `update.rs` with a new module that defines the update types
and implements the GitHub Releases API client.

**File:** `src/update.rs` (rewrite)

**Types to define:**

```rust
/// Result of checking for updates
pub enum UpdateCheck {
    /// No update available (current version is latest or newer)
    UpToDate,
    /// A newer version is available
    Available {
        version: String,            // e.g., "0.2.2"
        asset_url: String,          // Direct download URL for the platform's binary
        checksum_url: String,       // Direct download URL for checksums.txt
        asset_name: String,         // e.g., "bursar-linux-x86_64"
    },
    /// Could not check (network error, API error, parse error)
    Failed(String),
}

/// Result of the full update process
pub enum UpdateResult {
    /// No update was needed
    UpToDate,
    /// Update downloaded and ready — caller should restart
    ReadyToRestart {
        new_exe_path: PathBuf,
        version: String,
    },
    /// Update check or download failed — continue with current version
    Failed(String),
}
```

**GitHub API client function:**

```rust
pub fn check_for_update(github_repo: &str) -> UpdateCheck
```

Implementation:
1. Build URL: `https://api.github.com/repos/{github_repo}/releases/latest`
2. GET with headers:
   - `User-Agent: bursar/{current_version}` (where current_version = `env!("CARGO_PKG_VERSION")`)
   - `Accept: application/vnd.github+json`
3. Handle non-200 responses → `UpdateCheck::Failed`
4. Parse JSON response body using `serde_json::Value` (no need for a full struct):
   - `tag_name` → strip leading `v`, parse with `semver::Version`
   - `assets` array → find the entry whose `name` matches the current platform's asset name
   - Same `assets` array → find the entry whose `name` is `"checksums.txt"`
   - Extract `browser_download_url` from each matched asset
5. Compare versions: if remote > local → `UpdateCheck::Available`, else → `UpdateCheck::UpToDate`

**Platform asset name** (compile-time constant):

```rust
#[cfg(all(target_os = "linux", target_arch = "x86_64"))]
const PLATFORM_ASSET_NAME: &str = "bursar-linux-x86_64";

#[cfg(all(target_os = "windows", target_arch = "x86_64"))]
const PLATFORM_ASSET_NAME: &str = "bursar-windows-x86_64.exe";
```

**ureq timeout:** Use a 10-second timeout for the API check (shorter than the 120s AI
timeout — this should be fast). If it times out, return `UpdateCheck::Failed`.

**Tests:**
- Parse a sample GitHub API JSON response and extract the correct asset URLs
- Version comparison: `0.2.0` < `0.2.1`, `0.2.1` == `0.2.1`, `0.3.0` > `0.2.1`
- Missing asset for platform returns `Failed`
- Missing `checksums.txt` asset returns `Failed`

**Verification:**
- `cargo fmt && cargo clippy -D warnings && cargo test` all pass

**Commit:** `V2.2 Phase 2, Task 1: rewrite update module with types and GitHub API client`

### Task 2: Pre-flight Checks

**What:** Implement the checks that run after an update is detected but before any download
begins: symlink detection and write permission verification.

**File:** `src/update.rs`

**Function:**

```rust
/// Run pre-flight checks to verify the binary can be replaced.
/// Returns Ok(exe_path) if checks pass, Err(reason) if not.
pub fn preflight_check() -> Result<PathBuf, String>
```

Implementation:

1. **Get binary path:** `std::env::current_exe()` → if this fails, return error

2. **Symlink check:** Use `std::fs::symlink_metadata()` on the path from `current_exe()`.
   Check `metadata.file_type().is_symlink()`.

   **Important nuance:** `current_exe()` on Linux resolves through `/proc/self/exe` which
   itself is a symlink, but it returns the resolved real path. The check should be:
   get the parent directory + the binary filename, and check if THAT path is a symlink.
   Alternatively, compare `current_exe()` result against `std::env::args().next()` resolved
   via `std::fs::canonicalize()` — if they differ, a symlink is involved.

   The simplest reliable approach: get `current_exe()` result (already resolved), then
   check `symlink_metadata()` on it. Since it's already resolved, `is_symlink()` will be
   false. So instead: check `symlink_metadata()` on the argv[0] path or on the known
   install path. **Ask First** if uncertain about the detection approach.

3. **Write permission check:** Create a temporary file in the binary's parent directory
   (e.g., `.bursar-update-check-{pid}`), then immediately delete it. If creation fails,
   return `Err("permission denied on {dir}...")`

4. Return `Ok(exe_path)` if all checks pass.

**Tests:**
- Returns `Ok` when pointed at a writable directory (use a temp dir)
- Detects read-only directories (create temp dir, remove write permission, check, restore)

**Note:** Symlink detection is hard to unit test portably. A code comment explaining the
logic is sufficient. Opus will review the detection approach carefully.

**Verification:**
- `cargo fmt && cargo clippy -D warnings && cargo test` all pass

**Commit:** `V2.2 Phase 2, Task 2: implement pre-flight checks for update safety`

### Task 3: Download with Progress Callback

**What:** Implement the chunked binary download that reports progress back to the caller
for rendering the progress bar.

**File:** `src/update.rs`

**Function:**

```rust
pub fn download_with_progress<F>(
    url: &str,
    dest: &Path,
    mut on_progress: F,
) -> Result<(), String>
where
    F: FnMut(u64, Option<u64>),
```

Implementation:
1. `ureq::get(url)` with `User-Agent` and `Accept: application/octet-stream` headers
2. Read `Content-Length` header → `Option<u64>`
3. Create destination file
4. Read response body in 8 KB chunks
5. After each chunk: write to file, increment counter, call `on_progress`
6. After completion: verify size matches `Content-Length` if present
7. On any error: delete the partial file before returning

**Timeout:** 300 seconds via `ureq::AgentBuilder::timeout()`.

**Tests:**
- Progress callback receives incrementing byte counts (test with in-memory data)
- Partial file cleanup on error

**Verification:**
- `cargo fmt && cargo clippy -D warnings && cargo test` all pass

**Commit:** `V2.2 Phase 2, Task 3: implement chunked download with progress callback`

### Task 4: SHA256 Verification

**What:** Download and parse the checksums file, then verify the downloaded binary's hash.

**File:** `src/update.rs`

**Functions:**

```rust
pub fn fetch_expected_checksum(checksum_url: &str, asset_name: &str) -> Result<String, String>
pub fn verify_checksum(file_path: &Path, expected_hex: &str) -> Result<(), String>
```

**`fetch_expected_checksum`:** GET the checksums.txt, parse lines of format
`{hex_hash}  {filename}` (two spaces, matching `sha256sum` output), find matching asset,
return hex hash.

**`verify_checksum`:** Open file, create `sha2::Sha256` hasher, read in 8 KB chunks,
finalize, hex-encode result (iterate bytes with `format!("{:02x}")`), compare case-insensitive.
Mismatch → `Err` with both hashes for debugging.

**Tests:**
- Parse multi-entry checksums.txt, extract correct hash
- Asset not found in checksums → error
- SHA256 of `b"hello world"` = `b94d27b9934d3e08a52e52d7da7dabfac484efe37a5380ee9088f7ace2efcde9`
- Wrong hash produces error with both hashes

**Verification:**
- `cargo fmt && cargo clippy -D warnings && cargo test` all pass

**Commit:** `V2.2 Phase 2, Task 4: implement SHA256 checksum verification`

### Task 5: Binary Replacement and Restart

**What:** Implement the platform-specific binary swap and process restart.

**File:** `src/update.rs`

**Functions:**

```rust
/// Does not return on success. On failure, attempts rollback and returns error.
pub fn replace_and_restart(
    current_exe: &Path,
    new_binary: &Path,
    terminal: &mut Terminal<CrosstermBackend<Stdout>>,
) -> Result<(), String>

/// Clean up .old binary from previous update. Silent on failure.
pub fn cleanup_old_binary()
```

**`replace_and_restart`:**
1. `old_path` = current exe path + `.old` appended to filename
2. Rename current → old. On failure: return error (nothing changed)
3. Rename new → current. On failure: restore old → current, return error
4. **Linux** (`#[cfg(target_os = "linux")]`): set permissions 0o755, `exec()` with same args
5. **Windows** (`#[cfg(target_os = "windows")]`): restore terminal (disable raw mode, leave
   alternate screen) FIRST, then `spawn()` new process, then `process::exit(0)`

The `terminal` parameter is needed for Windows cleanup. On Linux, suppress the unused
warning appropriately.

**`cleanup_old_binary`:** Get `current_exe()`, construct `.old` path, `remove_file()`,
ignore all errors.

**Tests:**
- `cleanup_old_binary` with no `.old` file → no panic
- Rename sequence in temp dir with mock files → verify file positions
- Rollback: simulate step 3 failure, verify old is restored

**Note:** `exec()` / `spawn()+exit()` cannot be unit tested. Rename logic is the testable
part. Restart verified manually by developer.

**Verification:**
- `cargo fmt && cargo clippy -D warnings && cargo test` all pass

**Commit:** `V2.2 Phase 2, Task 5: implement binary replacement and restart`

### Task 6: Wire Update Pipeline into Main

**What:** Orchestrate the full update flow in `main.rs`: cleanup → check → pre-flight →
download with progress → verify → replace → restart.

**File:** `src/main.rs`

**Changes:**

1. **Early in `main()`**: call `update::cleanup_old_binary()`

2. **In the `Splash` state**, after terminal init: run the update pipeline. Extract the
   nested match chain into a helper function to keep `main()` readable:

   ```rust
   /// Attempt the full update flow. Returns Ok(()) if no update needed or update
   /// succeeded (process replaced — doesn't actually return). Returns Err(warning)
   /// if update failed and we should continue with current version.
   fn attempt_update(
       github_repo: &str,
       splash: &mut SplashState,
       terminal: &mut Terminal<CrosstermBackend<Stdout>>,
   ) -> Result<(), String>
   ```

   The helper function orchestrates: `check_for_update` → `preflight_check` → set
   splash status + progress → `download_with_progress` (with closure that updates
   `splash.progress` and calls `terminal.draw()`) → `fetch_expected_checksum` →
   `verify_checksum` → set Complete → `replace_and_restart`.

   On any `Err`, clean up the downloaded file if it exists and return the error string.

3. In `main()`, call the helper. If it returns `Err(warning)`, pass as `update_notice`
   to `StartupScreen`. If it returns `Ok(())`, proceed normally.

4. **Splash minimum display time**: only apply the 1-second minimum when no update is
   happening. If an update downloads, the download time replaces the wait.

**Important:** The progress callback closure captures `&mut splash` and calls
`terminal.draw()`. Same forced-render pattern as `ai_handler.rs`. Works because the
callback runs between chunk reads in the same thread.

**Verification:**
- App launches normally with no update available
- App launches normally with no internet (no crash, no error)
- Warning shows on startup screen with nonexistent repo (simulated failure)
- `cargo fmt && cargo clippy -D warnings && cargo test` all pass

**Commit:** `V2.2 Phase 2, Task 6: wire update pipeline into main splash flow`
