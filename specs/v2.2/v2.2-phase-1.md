### Task 4: Add Progress Bar Area to Splash Screen

**What:** Add the rendering infrastructure for the update progress bar on the splash screen.
The actual download logic comes in Phase 2 — this task just adds the visual elements and
the state needed to drive them.

**File:** `src/main.rs`

**Changes:**

1. Add state fields to the splash screen state (or a new struct if needed):
   ```rust
   update_status: Option<String>,    // e.g., "Updating to v0.2.2. . ."
   progress: Option<UpdateProgress>, // None = no download in progress
   ```

2. Define `UpdateProgress`:
   ```rust
   enum UpdateProgress {
       Determinate { percent: u8 },         // 0-100, shows ▰▰▰▱▱▱ [40%]
       Indeterminate,                        // shows ▰▱▰▱▰▱▰▱▰▱ (no Content-Length)
       Complete,                             // shows ▰▰▰▰▰▰▰▰▰▰ [100%]
   }
   ```

3. Add a render function for the progress bar:
   - Characters: `▰` (U+25B0) for filled, `▱` (U+25B1) for unfilled
   - Bar width: 10 characters
   - Determinate: `▰▰▰▰▱▱▱▱▱▱ [40%]` — filled count = `percent / 10`
   - Indeterminate: `▰▱▰▱▰▱▰▱▰▱` — static alternating pattern
   - Complete: `▰▰▰▰▰▰▰▰▰▰ [100%]`
   - Both the status text and progress bar are centered beneath the version number

4. Wire into the splash render: if `update_status` is `Some`, render the status text;
   if `progress` is `Some`, render the progress bar below it. If both are `None`,
   the splash looks exactly as it does today (just with the centered version from Task 3).

**Verification:**
- Unit test: progress bar rendering produces correct strings for 0%, 50%, 100%, and
  indeterminate
- Visual: the splash screen still looks correct with no update in progress (no regressions)
- `cargo fmt && cargo clippy -D warnings && cargo test` all pass

**Commit:** `V2.2 Phase 1, Task 4: add progress bar rendering to splash screen`

### Task 5: Create GitHub Actions Release Workflow

**What:** Create the CI/CD workflow file that builds release binaries for Linux and Windows
when a version tag is pushed.

**File:** `.github/workflows/release.yml`

**Workflow structure:**

```yaml
name: Release

on:
  push:
    tags:
      - 'v*'

jobs:
  build-linux:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
        with:
          components: rustfmt, clippy
      - run: cargo fmt --check
      - run: cargo clippy --all-targets --all-features -- -D warnings
      - run: cargo test
      - run: cargo build --release
      - run: mv target/release/bursar bursar-linux-x86_64
      - uses: actions/upload-artifact@v4
        with:
          name: bursar-linux-x86_64
          path: bursar-linux-x86_64

  build-windows:
    runs-on: windows-latest
    steps:
      - uses: actions/checkout@v4
      - uses: dtolnay/rust-toolchain@stable
        with:
          components: rustfmt, clippy
      - run: cargo fmt --check
      - run: cargo clippy --all-targets --all-features -- -D warnings
      - run: cargo test
      - run: cargo build --release
      - run: Move-Item -Path target\release\bursar.exe -Destination bursar-windows-x86_64.exe
      - uses: actions/upload-artifact@v4
        with:
          name: bursar-windows-x86_64
          path: bursar-windows-x86_64.exe

  release:
    needs: [build-linux, build-windows]
    runs-on: ubuntu-latest
    permissions:
      contents: write
    steps:
      - uses: actions/download-artifact@v4
        with:
          name: bursar-linux-x86_64
      - uses: actions/download-artifact@v4
        with:
          name: bursar-windows-x86_64
      - name: Generate checksums
        run: sha256sum bursar-linux-x86_64 bursar-windows-x86_64.exe > checksums.txt
      - uses: softprops/action-gh-release@v2
        with:
          files: |
            bursar-linux-x86_64
            bursar-windows-x86_64.exe
            checksums.txt
          generate_release_notes: true
```

**Notes:**
- The `permissions: contents: write` on the release job is required for creating GitHub
  Releases
- `dtolnay/rust-toolchain@stable` is the standard way to install Rust in CI
- `softprops/action-gh-release@v2` auto-creates the release from the tag and attaches files
- `generate_release_notes: true` uses GitHub's auto-generated notes from commits
- Windows uses `Move-Item` (PowerShell) instead of `mv` for reliability on GitHub Actions
  Windows runners

**Constraints:**
- This file cannot be tested locally by Sonnet. The developer will test it by pushing a
  tag after Phase 1 review.
- Do NOT create any other workflow files (e.g., CI on push to main). Only the release
  workflow for now.

**Verification:**
- YAML is valid (no syntax errors)
- File is at the correct path: `.github/workflows/release.yml`
- `cargo fmt && cargo clippy -D warnings && cargo test` all pass (no Rust changes in
  this task, but verify nothing is broken)

**Commit:** `V2.2 Phase 1, Task 5: create GitHub Actions release workflow`
