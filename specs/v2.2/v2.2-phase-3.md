# V2.2 Phase 3: In-App Feedback — Bug Reports & Feature Requests

## Overview

This phase adds the ability for users to submit bug reports and feature requests directly
from the `?` help overlay. A multi-line text input modal collects the user's description,
then Bursar opens the default browser with a pre-filled GitHub issue containing the
description, system information, and recent audit log entries.

## Tasks

### Task 1: Create FeedbackModal Widget

**What:** Build a multi-line text input modal for collecting bug reports and feature
request descriptions.

**File:** `src/widgets/feedback_modal.rs` (new)

**Types:**

```rust
pub enum FeedbackType {
    Bug,
    Feature,
}

pub enum FeedbackAction {
    Submit(FeedbackType, String),  // type + full text content (lines joined with \n)
    Cancel,
    None,
}

pub struct FeedbackModal {
    feedback_type: FeedbackType,
    lines: Vec<String>,           // text buffer, one entry per line
    cursor_row: usize,
    cursor_col: usize,
    scroll_offset: usize,         // for scrolling when content exceeds visible area
}
```

**Constructor:**
```rust
impl FeedbackModal {
    pub fn new(feedback_type: FeedbackType) -> Self
}
```

Initialize with a single empty line, cursor at (0, 0).

**Key handling:**
```rust
pub fn handle_key(&mut self, key: KeyEvent) -> FeedbackAction
```

| Key | Action |
|-----|--------|
| Printable char | Insert at cursor position, advance cursor |
| Backspace | Delete char before cursor; if at line start, merge with previous line |
| Delete | Delete char at cursor; if at line end, merge with next line |
| Enter | Split current line at cursor, create new line below, move cursor to start of new line |
| ← / → | Move cursor left/right; wrap to previous/next line at boundaries |
| ↑ / ↓ | Move cursor up/down; clamp column to line length |
| Home | Move cursor to start of current line |
| End | Move cursor to end of current line |
| Ctrl+S | Return `FeedbackAction::Submit` with type and joined text |
| Esc | Return `FeedbackAction::Cancel` |

**Rendering:**
```rust
pub fn render(&self, frame: &mut Frame, area: Rect)
```

Layout:
- Use `centered_rect(60, 40, area)` for the modal (60% width, 40% height)
- Title bar: "Describe the bug:" or "Describe the feature:" depending on type
- Text area fills the center (with 1-char padding on each side)
- Bottom bar: `Ctrl+S: Submit | Esc: Cancel` (rendered as a `Paragraph` with
  `Alignment::Center`, styled in the same key hint style as other modals)
- Cursor rendered using `frame.set_cursor_position()` at the correct row/col
- If content exceeds visible rows, scroll to keep cursor visible (adjust `scroll_offset`)

**Line wrapping:** Visual wrapping at the widget boundary. Each logical line in `self.lines`
may span multiple display rows. For simplicity in V2.2, long lines that exceed the widget
width can simply be truncated at the display boundary (the user will see the beginning of
the line). Full visual line wrapping would add significant complexity for minimal benefit —
users writing multi-paragraph text will finish in the browser anyway.

**Export to `src/widgets/mod.rs`:** Add `pub mod feedback_modal;` and re-export the
public types.

**Tests:**
- Insert characters, verify buffer content
- Enter creates new line and splits correctly
- Backspace at line start merges with previous line
- Cursor movement: up/down clamping, left/right wrapping at boundaries
- Submit returns joined text with `\n` separators
- Cancel returns `FeedbackAction::Cancel`

**Verification:**
- `cargo fmt && cargo clippy -D warnings && cargo test` all pass

**Commit:** `V2.2 Phase 3, Task 1: create FeedbackModal multi-line text input widget`

### Task 2: URL Construction and Browser Launch

**What:** Build the GitHub issue URL from the user's description and system info, and
launch the default browser.

**File:** `src/widgets/feedback_modal.rs` (add functions, or a new helper module — keep it
co-located with the modal since the URL is built from the modal's output)

**Functions:**

```rust
/// Build a complete GitHub issue URL with pre-filled title, body, and labels.
pub fn build_issue_url(
    feedback_type: &FeedbackType,
    description: &str,
    entity_name: Option<&str>,
    recent_audit_entries: &[String],  // pre-formatted audit log lines
) -> String

/// Open a URL in the default browser. Returns Ok(()) on success, Err on failure.
pub fn open_in_browser(url: &str) -> Result<(), String>
```

**`build_issue_url` implementation:**

1. Determine the base URL: `https://github.com/trout-dev-fwd/bursar/issues/new`

2. Build the **title**: extract the first line of `description`
   - Bug: `[Bug] {first_line}` (truncate to 100 chars if needed)
   - Feature: `[Feature] {first_line}`

3. Build the **body** from template:

   For bugs:
   ```
   {description}

   ---

   **System Information**
   - Bursar version: {env!("CARGO_PKG_VERSION")}
   - OS: {std::env::consts::OS} {std::env::consts::ARCH}
   - Entity: {entity_name or "None"}

   **Recent Audit Log**
   ```
   {audit_entries joined with newlines}
   ```
   ```

   For features:
   ```
   {description}

   ---

   **System Information**
   - Bursar version: {env!("CARGO_PKG_VERSION")}
   - OS: {std::env::consts::OS} {std::env::consts::ARCH}
   ```

   If `recent_audit_entries` is empty (no entity open), omit the audit log section
   entirely (even for bugs).

4. Determine **labels**: `"bug"` for Bug, `"enhancement"` for Feature

5. URL-encode all parameters using `urlencoding::encode()`:
   `{base}?title={encoded_title}&body={encoded_body}&labels={encoded_labels}`

6. **Length check:** If the total URL exceeds 8,000 characters, truncate the description
   portion and append `\n\n[Description truncated for URL length — please add details in the issue]`,
   then rebuild. Keep the system info and audit log intact (they're short).

**`open_in_browser` implementation:**

```rust
#[cfg(target_os = "linux")]
fn open_browser_cmd(url: &str) -> Result<(), String> {
    std::process::Command::new("xdg-open")
        .arg(url)
        .spawn()
        .map_err(|e| format!("Could not open browser: {e}"))?;
    Ok(())
}

#[cfg(target_os = "windows")]
fn open_browser_cmd(url: &str) -> Result<(), String> {
    std::process::Command::new("cmd")
        .args(["/c", "start", "", url])  // empty string for window title
        .spawn()
        .map_err(|e| format!("Could not open browser: {e}"))?;
    Ok(())
}
```

Fire-and-forget: spawn and don't wait. If spawn fails, return the error message.

**Tests:**
- Bug URL includes `labels=bug`, title starts with `[Bug]`
- Feature URL includes `labels=enhancement`, title starts with `[Feature]`
- URL includes system info (version, OS, arch)
- Bug with audit entries includes them in the body
- Feature with audit entries does NOT include them
- Bug with empty audit entries omits the section
- URL exceeding 8000 chars gets truncated with the truncation notice
- Title truncates to 100 chars if first line is long
- All special characters in description are properly URL-encoded

**Verification:**
- `cargo fmt && cargo clippy -D warnings && cargo test` all pass

**Commit:** `V2.2 Phase 3, Task 2: implement URL construction and browser launch`

### Task 4: Wire Feedback into App — Key Dispatch and State

**What:** Connect the `b` and `f` keys in the help overlay to the FeedbackModal, handle
the modal's lifecycle, and process submissions.

**Files:** `src/app/mod.rs`, `src/app/key_dispatch.rs`

**Changes to `src/app/mod.rs`:**

Add field to `App`:
```rust
feedback_modal: Option<FeedbackModal>,
```

Initialize as `None` in `App::new()`.

**Changes to `src/app/key_dispatch.rs`:**

1. **In the help overlay key handler** (the block that runs when `self.show_help == true`):
   - On `b` key: set `self.show_help = false`, set `self.feedback_modal = Some(FeedbackModal::new(FeedbackType::Bug))`
   - On `f` key: set `self.show_help = false`, set `self.feedback_modal = Some(FeedbackModal::new(FeedbackType::Feature))`

2. **Add feedback modal to the key dispatch priority chain.** It should be checked
   AFTER the help overlay and user guide, but BEFORE the file picker, import wizard,
   and chat panel. Insert a new priority level:

   ```rust
   // In the key dispatch chain:
   if let Some(ref mut modal) = self.feedback_modal {
       match modal.handle_key(key) {
           FeedbackAction::Submit(feedback_type, description) => {
               // Fetch last 5 audit entries
               let audit_entries: Vec<String> = self.entity.db.audit()
                   .list_recent(5)  // may need to add this method
                   .unwrap_or_default()
                   .iter()
                   .map(|e| format!("{} | {} | {}", e.created_at, e.action_type, e.description))
                   .collect();

               let url = build_issue_url(
                   &feedback_type,
                   &description,
                   Some(&self.entity.name),
                   &audit_entries,
               );

               match open_in_browser(&url) {
                   Ok(()) => self.status_bar.set_message(
                       match feedback_type {
                           FeedbackType::Bug => "Bug report opened in browser",
                           FeedbackType::Feature => "Feature request opened in browser",
                       }
                   ),
                   Err(e) => self.status_bar.set_error(format!(
                       "{}. File manually at https://github.com/trout-dev-fwd/bursar/issues",
                       e
                   )),
               }
               self.feedback_modal = None;
           }
           FeedbackAction::Cancel => {
               self.feedback_modal = None;
           }
           FeedbackAction::None => {}
       }
       return;  // modal consumed the key
   }
   ```

3. **Render the feedback modal.** In the App render function, if `feedback_modal` is `Some`,
   render it on top of everything (same layer as other modal overlays). It should render
   AFTER the main tab content but BEFORE the help overlay (since pressing `b`/`f` closes
   the help overlay first, the feedback modal and help overlay should never be visible
   simultaneously — but defensive ordering doesn't hurt).

**AuditRepo change (if needed):**

The existing `AuditRepo::list()` may not have a "last N entries" method. If not, add:

```rust
pub fn list_recent(&self, limit: usize) -> Result<Vec<AuditEntry>>
```

Simple query: `SELECT * FROM audit_log ORDER BY created_at DESC LIMIT ?1`

If `list()` already supports this via filter parameters, use that instead. Check the
existing API before adding a new method.

**Tests:**
- `b` key when help overlay is open closes overlay and opens FeedbackModal with Bug type
- `f` key when help overlay is open closes overlay and opens FeedbackModal with Feature type
- `b` key when help overlay is NOT open does NOT open the feedback modal
- Esc in feedback modal closes it and sets `feedback_modal = None`
- Submit in feedback modal closes it (browser launch is hard to test — verify the URL
  construction in Task 2 tests instead)

**Verification:**
- Visual: press `?`, then `b` — modal appears with "Describe the bug:" title
- Visual: type text, press `Ctrl+S` — browser opens with pre-filled issue
- Visual: press `?`, then `f` — modal appears with "Describe the feature:" title
- `cargo fmt && cargo clippy -D warnings && cargo test` all pass
