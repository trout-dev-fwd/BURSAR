## Design Decisions (Locked)

These decisions were made during planning and are not open for re-evaluation:

1. **Two-phase architecture (Option B):** The startup screen is a separate struct, NOT a mode inside `App`. A top-level `AppState` enum in `main.rs` owns the state machine: `Splash → Startup(StartupScreen) → Running(App)`. This avoids infecting `app.rs` with `Option<EntityContext>` checks.

2. **Three-state wrapper:** `main.rs` owns a loop that delegates render/key-handling to whichever `AppState` variant is active. `App::run` no longer owns the outermost loop — extract its loop body into methods (`render`, `handle_event`, `tick`) that the wrapper calls.

3. **`toml_edit` for workspace.toml writes:** Use the `toml_edit` crate (format-preserving TOML parser) for all entity management operations. This preserves comments, key ordering, and human-edited formatting. Add `toml_edit` to `Cargo.toml` dependencies. The existing `toml` + `serde` dependency stays for reads.

4. **1-second minimum splash:** The splash screen (logo + version + update check) always displays for at least 1 second, even if no `[updates]` section exists. This serves as a branding moment and gives the impression the software is initializing.

5. **`last_opened_entity` persisted:** Add a `last_opened_entity = "Entity Name"` field to workspace.toml (top-level). Use `toml_edit` to read/write it. The startup screen pre-selects this entity. Update it when an entity is opened.

6. **No return-to-picker:** In this version, once an entity is opened, the only way back is to quit (`q`) and relaunch. The `Running → Startup` transition is deferred.

7. **New `TextInputModal` widget:** Entity add and edit need a generic text-input prompt. Create `src/widgets/text_input_modal.rs` — title, text buffer, cursor position, returns `Some(String)` on Enter or `None` on Esc. Add to key dispatch priority (above tab dispatch, below confirmation modal).

## Task 2: Startup Screen + Splash

**Goal:** Add a three-state app wrapper with splash screen, startup/entity-picker screen, and transition to the existing app.

### Architecture

#### `AppState` enum (in `main.rs` or a new `src/app_state.rs`)

```rust
enum AppState {
    Splash,
    Startup(StartupScreen),
    Running(App),
}
```

#### Wrapper loop in `main.rs`

The wrapper owns the terminal and the event loop:

```
fn main():
    initialize terminal
    parse workspace config
    state = AppState::Splash

    loop:
        match state:
            Splash =>
                render splash (logo + version + "Checking for updates...")
                // Update check logic is Task 4 — for now, just sleep 1 second
                sleep(1s)
                state = Startup(StartupScreen::new(config))

            Startup(screen) =>
                terminal.draw(screen.render)
                poll for key event
                match screen.handle_key(key):
                    StartupAction::OpenEntity(index) =>
                        update last_opened_entity in workspace.toml
                        construct App with selected entity
                        state = Running(app)
                    StartupAction::Quit => break
                    StartupAction::Redraw => continue
                    _ => continue

            Running(app) =>
                // Delegate to existing App logic
                // App::run becomes App::tick + App::render + App::handle_event
                // When app.should_quit => break
```

#### Refactoring `App::run`

Currently `App::run` owns the event loop. Extract the loop body into:
- `App::render(&mut self, terminal: &mut Terminal)` — one frame
- `App::handle_event(&mut self, event: Event)` — one key/tick
- `App::tick(&mut self)` — periodic work (chat typewriter, status bar expiry, etc.)
- `App::should_quit(&self) -> bool`

The wrapper calls these in sequence. `App::run` can remain as a convenience that calls the wrapper, or be removed — whichever is cleaner. The goal is that `App` no longer needs to own the terminal or the loop.

You may be able to extend it with the new UI struct, or it may be cleaner to create `src/startup_screen.rs` for the TUI screen and keep `startup.rs` for initialization logic. Make the decision based on what's already in the file.

#### Startup screen rendering

Layout (top to bottom):
1. ASCII banner — centered horizontally, styled with accent color
2. Version — right-aligned below banner: `v{version}`
3. Update notice area — (Task 4, placeholder for now): if `update_notice.is_some()`, render it in yellow/green
4. Entity list — centered, each entry shows name and db path
   - Selected entity highlighted with `>>` marker or inverse colors
   - If list is empty: "No entities configured. Press 'a' to add one."
5. Hotkey bar at bottom: `[Enter] Open  [a] Add  [e] Edit  [d] Delete  [q] Quit`

#### Startup screen key handling

- `Up/k` / `Down/j`: Navigate entity list
- `Enter`: Return `StartupAction::OpenEntity(selected_index)`
- `a`: Return `StartupAction::AddEntity` (handled in Task 3, no-op for now)
- `e`: Return `StartupAction::EditEntity` (handled in Task 3, no-op for now)
- `d`: Return `StartupAction::DeleteEntity` (handled in Task 3, no-op for now)
- `q`: Return `StartupAction::Quit`

#### `last_opened_entity`

- On startup screen creation, read `last_opened_entity` from workspace.toml
- Set `selected_index` to the matching entity (by name), or 0 if not found
- When `OpenEntity` fires, use `toml_edit` to update `last_opened_entity` in workspace.toml

#### db_path resolution

Entity `db_path` values in workspace.toml may be relative. They must be resolved relative to the directory containing workspace.toml, NOT the current working directory. Check how the existing `startup.rs` resolves paths and follow the same strategy. The add-entity flow (Task 3) must also resolve paths consistently.

## Task 3: Entity Management

**Goal:** Implement add, edit, and delete for entities on the startup screen.

### New Widget: `TextInputModal`

Create `src/widgets/text_input_modal.rs`:

```rust
pub struct TextInputModal {
    title: String,
    buffer: String,
    cursor_pos: usize,
}

pub enum TextInputAction {
    Confirm(String),
    Cancel,
    None,
}
```

Features:
- Renders as a centered modal box (similar style to the existing confirmation modal)
- Title displayed at top of modal
- Single-line text input with visible cursor
- Left/Right/Home/End for cursor movement
- Backspace/Delete for text editing
- Enter → `TextInputAction::Confirm(buffer.clone())`
- Esc → `TextInputAction::Cancel`

Add to key dispatch in the startup screen: when a `TextInputModal` is active (`Option<TextInputModal>` on `StartupScreen`), all keys go to the modal first.

### Add Entity (`a`)

Flow:
1. Open `TextInputModal` with title "Entity name:"
2. On `Confirm(name)`:
   a. Derive database filename: lowercase name, spaces to hyphens, append `.sqlite`
      - Example: "My Farm LLC" → `my-farm-llc.sqlite`
   b. Derive entity toml filename: same stem, `.toml` extension
      - Example: "My Farm LLC" → `my-farm-llc.toml`
   c. Resolve the directory: use the same directory as the first existing entity's `db_path` (resolved relative to workspace.toml). If no entities exist, use the workspace.toml directory itself.
   d. Use `toml_edit` to append a new `[[entities]]` block to workspace.toml:
      ```toml
      [[entities]]
      name = "My Farm LLC"
      db_path = "my-farm-llc.sqlite"
      config_path = "my-farm-llc.toml"
      ```
   e. Create the entity `.toml` file with minimal defaults (empty or just a comment)
   f. Do NOT create the `.sqlite` file — `EntityDb::open` creates it on first open
   g. Refresh the entity list from workspace.toml

### Edit Entity (`e`)

Flow:
1. Open `TextInputModal` with title "Rename entity:" and pre-filled with current name
2. On `Confirm(new_name)`:
   a. Use `toml_edit` to update the `name` field of the selected `[[entities]]` entry
   b. Do NOT rename database or toml files
   c. Refresh the entity list

### Delete Entity (`d`)

Flow:
1. Show the existing confirmation modal: "Remove '{name}' from Bursar? The database file will NOT be deleted."
2. On confirm:
   a. Use `toml_edit` to remove the selected `[[entities]]` block from workspace.toml
   b. Do NOT delete the `.sqlite` or `.toml` files
   c. Show status message: "Removed. Database preserved at {db_path}"
   d. Refresh the entity list
   e. Adjust `selected_index` if it's now out of bounds

### `toml_edit` Patterns

Working with `[[entities]]` (arrays of tables) in `toml_edit`:

```rust
use toml_edit::{DocumentMut, Item, Table, Array, value};

// Parse
let content = std::fs::read_to_string(&path)?;
let mut doc = content.parse::<DocumentMut>()?;

// Read entities array
let entities = doc["entities"].as_array_of_tables();

// Append entity
let mut entity = toml_edit::Table::new();
entity["name"] = value("My Farm LLC");
entity["db_path"] = value("my-farm-llc.sqlite");
entity["config_path"] = value("my-farm-llc.toml");
doc["entities"].as_array_of_tables_mut()
    .expect("entities is array of tables")
    .push(entity);

// Remove entity at index
doc["entities"].as_array_of_tables_mut()
    .expect("entities is array of tables")
    .remove(index);

// Write back
std::fs::write(&path, doc.to_string())?;
```

Note: The above is illustrative. Check `toml_edit` docs for exact API — the array-of-tables API may differ slightly by version. Do NOT use `.unwrap()` — use proper error handling.

- The new entity's toml file is created but sqlite file is not

- All operations refresh the displayed list
- workspace.toml formatting is preserved (comments, key order) for untouched sections
- Startup screen works with zero entities (empty list, only `a` and `q` functional)
- All tests pass

## Task 4: Update Check

**Goal:** Check GitHub for newer releases and display a notice on the splash/startup screen.

### workspace.toml addition

```toml
[updates]
github_repo = "owner/bursar"  # optional — update check disabled if missing
```

If `[updates]` or `github_repo` is absent, skip the check entirely (no error).

### Implementation

#### In the splash phase of the wrapper loop:

```
Splash:
    render logo + version
    read [updates].github_repo from config
    if github_repo is set:
        render "Checking for updates..." below version
        force terminal draw
        result = check_for_update(github_repo, current_version)
        match result:
            Ok(Some(new_version)) => update_notice = Some(format!(...))
            Ok(None) => update_notice = None  // up to date
            Err(_) => update_notice = None    // silently skip
        // ensure at least 1 second has elapsed since splash started
        sleep remaining time
    else:
        sleep 1 second  // branding splash, no check
    state = Startup(StartupScreen::new(config, update_notice))
```

#### `check_for_update` function

```rust
fn check_for_update(github_repo: &str, current_version: &str) -> Result<Option<String>> {
    let url = format!("https://api.github.com/repos/{}/releases/latest", github_repo);
    let response = ureq::get(&url)
        .set("User-Agent", "bursar-update-check")  // GitHub requires User-Agent
        .timeout(std::time::Duration::from_secs(3))
        .call()?;
    let json: serde_json::Value = response.into_json()?;
    let tag = json["tag_name"].as_str().ok_or_else(|| anyhow!("no tag_name"))?;
    let remote = tag.strip_prefix('v').unwrap_or(tag);
    if is_newer(remote, current_version) {
        Ok(Some(remote.to_string()))
    } else {
        Ok(None)
    }
}
```

#### Version comparison: `is_newer(remote, local) -> bool`

```rust
fn is_newer(remote: &str, local: &str) -> bool {
    // Split on '.', parse each segment as u32
    // Compare left to right: if remote segment > local, return true
    // If remote segment < local, return false
    // If all equal, return false
    // If parsing fails for any segment, return false (silently skip)
}
```

Write unit tests for `is_newer`:
- `"1.2.0"` > `"1.1.0"` → true
- `"1.1.0"` > `"1.1.0"` → false
- `"1.1.0"` > `"1.2.0"` → false
- `"2.0.0"` > `"1.9.9"` → true
- `"1.0.1"` > `"1.0.0"` → true
- `"garbage"` vs anything → false

#### Update notice rendering

On the startup screen (not just splash), display below the version:

```
New version v1.2.0 available — github.com/owner/bursar/releases
```

Styled in yellow (`Color::Yellow`) to stand out from the rest of the screen.

#### Session caching

The `update_notice: Option<String>` is computed once during splash and passed to `StartupScreen`. Since there's no return-to-picker in this version, it's inherently session-cached.

### Verification

- With valid `[updates]` config and internet: update notice appears if remote version is newer
- With valid config but no internet: splash shows for ~3 seconds (timeout), no error displayed, transitions normally
- With no `[updates]` section: splash shows for 1 second, no check attempted
- Version comparison unit tests pass
- The splash screen renders the logo and "Checking for updates..." before the blocking call
- All tests pass
